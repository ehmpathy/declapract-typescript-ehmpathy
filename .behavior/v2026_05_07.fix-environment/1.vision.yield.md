# vision: migrate to sdk-environment + sdk-config

## the outcome world

### day-in-the-life with this in place

a developer adds a new service. they run `declapract fix` and get:

**sdk-environment:**
- `sdk-environment` dependency
- 12-line `src/utils/environment.ts` wrapper
- `stage` export (translates `prep` â†’ `dev` for serverless compat)

**sdk-config:**
- `sdk-config` + `zod` dependencies
- `src/utils/config/getConfig.ts` with `genGetConfig`
- `src/utils/config/config.schema.ts` with zod validation
- `config/prep.json` (renamed from `config/dev.json`)
- `$.at(aws::param/...)` syntax for SSM parameters

no more 167-line adhoc parser. no more `config-with-paramstore`. no more `__PARAM__` magic.

### before/after contrast

| aspect | before | after |
|--------|--------|-------|
| environment detection | 167-line adhoc parser | 12-line sdk-environment wrapper |
| config loader | config-with-paramstore | sdk-config with zod schema |
| SSM parameter syntax | `"__PARAM__"` (magic placeholder) | `"$.at(aws::param/org.svc.env.key)"` (explicit) |
| config file name | `config/dev.json` | `config/prep.json` |
| env var | `STAGE=dev\|prod` | `ACCESS=prep\|prod` |
| config types | untyped | zod schema validation |
| server format | `'AWS:LAMBDA'` enum | `'cloud@aws.lambda'` structured |
| commit tracker | none | `commit: 'v1.2.3@a1b2c3d'` |

### the "aha" moment

"wait, i don't need to understand how AWS IAM account alias detection works just to get my config loaded? and my config is typed now?"

## user experience

### usecases and goals

1. **get environment info** â€” know which resources to connect to
   ```ts
   import { getEnvironment } from 'sdk-environment';
   const env = await getEnvironment();
   // { access: 'prod', server: 'cloud@aws.lambda', commit: 'v1.2.3@a1b2c3d' }
   ```

2. **load config by environment** â€” get the right secrets
   ```ts
   import { getConfig } from '../config/getConfig';
   const config = await getConfig();
   // envStatic.config is 'prep' â†’ loads config/prep.json
   // zod validates, sdk-config resolves $.at() SSM params
   ```

3. **debug which version is running** â€” trace issues to commits
   ```ts
   log.info('starting', { commit: (await getEnvironment()).commit });
   ```

### timelines

- **upgrade extant repo**: run `declapract fix`, tests pass, done
- **new repo**: no action, comes pre-wired

## mental model

### how users describe this

"we use sdk-environment to detect what environment we're in â€” prod, prep, or test â€” so we load the right config and connect to the right databases."

### analogies

like `dotenv` but for runtime environment detection, not file loading.

### terms: theirs vs ours

| user says | sdk-environment term |
|-----------|---------------------|
| "which env am i in?" | `access` |
| "am i in lambda?" | `server` |
| "what version?" | `commit` |
| "dev environment" | `prep` (pre-production) |

## evaluation

### how well does it solve the goals?

| goal | score | notes |
|------|-------|-------|
| reduce boilerplate | excellent | 167 â†’ ~15 lines |
| eliminate drift | excellent | single package |
| simpler mental model | good | fewer concepts |
| backwards compat | good | `stage` export translates prepâ†’dev for serverless |

### pros

- single source of truth for environment detection
- automatic git commit tracking (new capability)
- structured server format enables better parsing
- pluggable parsers for custom needs
- test/prep/prod aligns with ehmpathy vocab

### cons

- breaking change: `dev` â†’ `prep` terminology (mitigated by `stage` export for serverless)
- breaking change: `STAGE` env var â†’ `ACCESS` env var
- breaking change: server format `'AWS:LAMBDA'` â†’ `'cloud@aws.lambda'`
- must update CI/CD pipelines to set `ACCESS=prep|prod` (via serverless accessByStage pattern)
- must rename `config/dev.json` â†’ `config/prep.json`
- must convert all `"__PARAM__"` â†’ `"$.at(aws::param/org.svc.env.key)"` with explicit paths
- must add `config.schema.ts` with zod schema
- must replace `config-with-paramstore` with `sdk-config`

### edgecases and pit-of-success

| edgecase | sdk-environment handling |
|----------|-------------------------|
| no AWS credentials | returns `null` for access, falls back to NODE_ENV |
| uncommitted changes | appends `+` to commit hash |
| unknown server | defaults to `'local@unix'` |

### src/utils/environment.ts (12 lines)

```ts
import { getEnvironment } from 'sdk-environment';

export { getEnvironment };

export const envStatic = getEnvironment.static();

export const stage = envStatic.access === 'prep' ? 'dev' : envStatic.access;
export const serviceClientStage = stage === 'prod' ? 'prod' : 'dev';
```

key patterns:
- `getEnvironment.static()` for synchronous access â€” no parsers needed
- `envStatic` exported for use in getConfig and other sync contexts
- `stage` export translates `prep` â†’ `dev` for serverless stage compat
- `serviceClientStage` for downstream service client SDKs

### src/utils/config/getConfig.ts (sdk-config)

```ts
import {
  genGetConfig,
  genSdkConfigSupplierAwsParameterStore,
} from 'sdk-config';
import { createCache } from 'simple-in-memory-cache';

import { envStatic } from '../environment';
import { configSchema } from './config.schema';

export const getConfig = genGetConfig({
  schema: configSchema,
  statics: 'config/*.json',
  cache: createCache({ expiration: { minutes: 5 } }),
  suppliers: [genSdkConfigSupplierAwsParameterStore()],
  environment: {
    config: envStatic.config,
    server: envStatic.server,
  },
});
```

key patterns:
- `sdk-config` replaces `config-with-paramstore`
- uses `envStatic.config` and `envStatic.server` for sync init
- zod schema for typed config validation
- SSM parameter supplier for AWS secrets

### config/*.json parameter syntax (sdk-config)

old (config/dev.json with config-with-paramstore):
```json
{
  "parameterStoreNamespace": "ahbode.svc-jobs.dev",
  "database": {
    "role": {
      "crud": {
        "password": "__PARAM__"
      }
    }
  }
}
```

new (config/prep.json with sdk-config):
```json
{
  "database": {
    "role": {
      "crud": {
        "password": "$.at(aws::param/ahbode.svc-jobs.dev.database.role.crud.password)"
      }
    }
  }
}
```

key patterns:
- `$.at(aws::param/{path})` is the SSM parameter resolution syntax
- explicit full parameter paths replace magic `__PARAM__` placeholder
- SSM paths preserved â€” prep.json still uses `.dev` SSM paths (no SSM migration)
- each `__PARAM__` â†’ `$.at(aws::param/{old-namespace}.{json.path.to.key})`
- enables static analysis of which parameters are needed
- old `parameterStoreNamespace` config key removed (namespace baked into paths)

## what is awkward?

### vocabulary mismatch

the old code uses `config` for "which config to load" and `access` for "which AWS account am i in".

sdk-environment uses `access` for "which resources can i reach" (test/prep/prod).

this is semantically cleaner but requires mental model update.

### the `dev` â†’ `prep` shift

ehmpathy vocabulary uses `prep` (pre-production) instead of `dev` (development). this aligns with `prod` symmetry but may surprise users who expect `dev`.

**config files renamed:** `config/dev.json` â†’ `config/prep.json`. envStatic.config returns `'prep'` and loads the renamed file.

**serverless stage NOT renamed:** lambdas remain deployed as `svc-xyz-dev`. the `stage` export translates `prep` â†’ `dev` for serverless compat.

### the `STAGE` â†’ `ACCESS` shift

CI/CD pipelines and serverless.yml files use `STAGE=dev|prod`. this must change to `ACCESS=prep|prod`.

**svc-jobs pattern:** serverless.yml uses `accessByStage` map:
```yaml
custom:
  accessByStage:
    dev: prep
    prod: prod
provider:
  environment:
    ACCESS: ${self:custom.accessByStage.${self:provider.stage}}
```

this affects:
- github actions workflows (should use ACCESS=prep|prod, not STAGE)
- serverless.yml environment variables (use accessByStage pattern)
- terraform variables
- any command that sets STAGE â†’ should set ACCESS instead

### server format change

old: enum `'CICD' | 'AWS:LAMBDA' | 'LOCAL'`
new: structured `'$tier@$platform'` strings

code that checks `if (env.server === 'AWS:LAMBDA')` must change to exact match.

**svc-jobs pattern** (from PR #182):
```ts
// before
if (environment.server === 'AWS:LAMBDA') { ... }

// after
if (environment.server === 'cloud@aws.lambda') { ... }
```

**found usage:** `src/practices/persist-with-rds/best-practice/src/utils/database/getDatabaseConnection.ts:50` checks `environment.server === 'AWS:LAMBDA'`. this must be updated to `environment.server === 'cloud@aws.lambda'`.

### the `$.at()` SSM syntax migration

this is significant manual work per repo:

**before (config/dev.json):**
```json
{
  "parameterStoreNamespace": "ahbode.svc-jobs.dev",
  "database": {
    "role": {
      "crud": { "password": "__PARAM__" }
    }
  }
}
```

**after (config/prep.json):**
```json
{
  "database": {
    "role": {
      "crud": { "password": "$.at(aws::param/ahbode.svc-jobs.dev.database.role.crud.password)" }
    }
  }
}
```

key points:
- file renamed `dev.json` â†’ `prep.json`, but SSM paths stay `$org.$service.dev.*`
- no SSM parameter migration needed â€” paths are preserved
- `parameterStoreNamespace` key removed (namespace baked into explicit paths)
- each `__PARAM__` â†’ `$.at(aws::param/{old-namespace}.{json.path.to.key})`

### Stage enum removal

old: `import { Stage } from '../environment'; stage === Stage.PRODUCTION`
new: string literal comparison `stage === 'prod'`

**svc-jobs pattern** (from PR #182):
```ts
// before (in sendSlackWebhookNotification.ts)
if (stage === Stage.PRODUCTION) { ... }

// after
if (stage === 'prod') { ... }
```

the `Stage` enum is no longer needed â€” use string literals directly.
