# blocker.1: check function throws plain Error instead of semantic error class

**rule**: .agent/repo=ehmpathy/role=mechanic/briefs/practices/code.prod/pitofsuccess.errors/rule.require.failloud.md

**locations**:
- src/practices/runtime-type-checking/bad-practices/joi-imports/src/**/*.ts.declapract.ts

The check function in joi-imports uses `throw new Error('does not import from joi')` which is a generic Error without context. According to the failloud rule, errors should use proper classes: ConstraintError (caller must fix, exit 2) or MalfunctionError (server must fix, exit 1). This error indicates a constraint violation (the file does not import from joi, so the user must fix it), so it should be ConstraintError with a hint. The current implementation hides the severity and prevents automated retry decisions.

**snippet**:
```ts
throw new Error('does not import from joi');
```
