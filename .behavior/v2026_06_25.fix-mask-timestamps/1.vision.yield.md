# 1.vision — fix: mask-timestamps husky guard

## the wish (restated)

reintroduce a husky pre-commit guard that **blocks any commit of a file that contains a timestamp** (`T?HH:MM:SS{timezone}`), **unless** the file is a `.ts` or `.sh` file.

on detection, halt with a constraint error:

> `timestamps are forbidden in snapshots. mask them to prevent permadrift`

---

## the outcome world

### day-in-the-life

a mechanic runs a test suite. a jest snapshot captures rendered cli output that embeds a live timestamp (`2026-06-25T[HH:MM:SS]Z`). they `git add` the `.snap` file and `git commit`.

**before** (no guard): the commit succeeds. the snapshot is now pinned to a wall-clock instant that will never recur. the next test run regenerates a *different* timestamp → snapshot diff → red ci → resnap → commit → repeat. the snapshot **permadrifts**: it can never be "correct" twice. worse, the noise hides real regressions.

**after** (guard in place): the commit halts immediately:

```
✋ timestamps are forbidden in snapshots. mask them to prevent permadrift
   └─ src/cli/render.test.ts.snap:42  →  2026-06-25T[HH:MM:SS]Z
```

the mechanic masks the timestamp in their snapshot serializer (e.g. `[timestamp]`), re-snaps, and commits clean. the snapshot is now stable forever.

### before/after contrast

| aspect | before | after |
|---|--------|-------|
| live timestamp lands in a `.snap` | committed silent | blocked at pre-commit |
| ci stability | flaky, perma-red on time | deterministic |
| feedback latency | found in ci minutes later | caught locally in seconds |
| `.ts` / `.sh` source with real time logic | n/a | allowed (not snapshots) |

### the "aha" moment

the value clicks the first time a developer watches a snapshot go red *for no code change* — purely because time moved forward. the guard converts that future-pain into a present, local, actionable halt: "mask it now, never drift again."

## user experience

### usecases & goals

- **mechanic**: commit test snapshots without accidental capture of wall-clock time → goal: deterministic, low-noise test suites.
- **reviewer**: trust that snapshot diffs reflect intent, not clock movement → goal: meaningful PR review.
- **repo maintainer (consumer repo)**: inherit this guard automatically via declapract → goal: org-wide pit-of-success without per-repo setup.

### contract inputs & outputs

this is a **git pre-commit hook**, so the "contract" is the commit attempt itself.

**input**: the set of staged files (`git diff --cached --name-only`), filtered to drop `.ts` and `.sh`.

**output**:
- exit `0` + silent (or a calm pass note) when no timestamps found.
- exit non-zero (constraint, exit code `2` per `rule.require.exit-code-semantics`) + the constraint message + the matched `file:line → match` list when a timestamp is found.

### what use of it looks like

```sh
$ git add src/render.test.ts.snap
$ git commit -m "test(render): snapshot cli output"

✋ timestamps are forbidden in snapshots. mask them to prevent permadrift
   └─ src/render.test.ts.snap:42  →  2026-06-25T[HH:MM:SS]Z

# developer masks the timestamp, then:
$ git add src/render.test.ts.snap
$ git commit -m "test(render): snapshot cli output"
[main abc1234] test(render): snapshot cli output
```

### timeline

1. dev stages files
2. `git commit` fires `.husky/pre-commit`
3. pre-commit sources `check.timestamps.sh` (alongside the current `check.yalc.sh`)
4. check scans staged non-`.ts`/`.sh` files
5. found → halt with constraint message; not found → proceed to the rest of the checks

## mental model

### how a user describes it to a friend

> "our repo won't let you commit a snapshot with a real clock-time in it — it makes you mask it first, so your tests don't go red every time the clock ticks."

### analogies / metaphors

- **a watermark detector**: stops "wet ink" (live timestamps) from a set into permanent record.
- **permadrift = a clock you can never set right**: every glance shows a new time, so the snapshot is never "the truth" twice.

### terms: theirs vs ours

| they say | we say |
|----------|--------|
| "flaky snapshot" | permadrift |
| "fake the time" | mask the timestamp |
| "git hook that yells at me" | pre-commit constraint guard |
| "the time-cop" | `check.timestamps.sh` |

## evaluation

### how well it solves the goal

directly. permadrift's root cause is unmasked timestamps that reach version control. a pre-commit guard is the earliest, cheapest interception point — local, fast, before the noise ever enters history.

### pros

- catches the problem at the source (pit-of-success, `rule.require.solve-at-cause`).
- inherited automatically by every consumer repo via declapract.
- no runtime cost, no dependency added — pure shell + git.
- the `.ts`/`.sh` carve-out keeps legitimate time logic & scripts unblocked.

### cons / costs

- **false positives**: any non-`.ts`/`.sh` file with a legit `HH:MM:SS` (a changelog, a fixture, a doc example) gets blocked. needs an escape hatch or precise regex.
- **regex fidelity**: `T?HH:MM:SS{timezone}` must be tight enough to dodge version strings, durations, or coordinates, yet loose enough to catch ISO variants.
- **bypass**: `git commit --no-verify` sidesteps it (acceptable — guard, not gate).

### edgecases & pit-of-success

| edgecase | outcome |
|----------|---------|
| `.ts` / `.sh` file with timestamps | allowed by design |
| `.snap` with masked `[timestamp]` | passes (no raw match) |
| markdown doc with `[HH:MM:SS]` example | **blocked** — open question: escape hatch? |
| ISO with `Z`, `+00:00`, `-05:00`, or none | regex must cover optional tz |
| time without seconds (`14:30`) | out of scope per wish (`HH:MM:SS` required) |
| staged deletion | name-only filter must skip removed files |
| binary files | should be skipped (text grep only) |
| **this repo's own `.md` artifacts** (dogfood) | **the route docs and this very vision hold `2026-06-25T[HH:MM:SS]Z` and are `.md`, not `.ts`/`.sh` — under a deny-list scope the guard would block a commit of its own definition.** decisive evidence for the deny-vs-allow-list fork (open question #1) |

## open questions & assumptions

### assumptions made

1. **the guard lives in the canonical husky practice** at `src/practices/husky/best-practice/.husky/check.timestamps.sh`, sourced from `pre-commit`, as a peer of the current `check.yalc.sh` pattern — and also applies to this repo's own root `.husky/`.
2. **"snapshots" is the motivation but the guard is file-type-based**, not snapshot-specific: it blocks the *pattern* in *any* non-`.ts`/`.sh` file. (the wish's carve-out is by extension, not by `.snap`.)
3. **only staged files** are scanned (pre-commit scope), not the whole tree.
4. **constraint semantics**: exit code `2` (caller-must-fix), message verbatim as wished.
5. **scope = detect, not mask**: this guard only **detects** a raw timestamp and halts. it does **not** mask anything. the mask mechanism (e.g. a jest snapshot serializer that rewrites a timestamp to `[timestamp]`) lives **consumer-side**, out of this guard's scope. the day-in-the-life above shows a consumer's mask step for context only — i did not verify any such serializer is present in a consumer repo.

### questions — triaged

1. **scope of the allow-list** → ✅ **DECIDED: default deny**. block all files *except* `.ts`/`.sh` (the deny-list-by-exclusion read). consequence accepted: this repo's own `.md` route docs and extensionless husky hooks fall under the net unless separately handled — see the dogfood + extensionless-shell notes in edgecases; execution must account for them (e.g. the guard's own `.behavior/**` docs may need an allowance, tied to q2).

## what is awkward?

1. **extension-based allow-list is coarse**: dropped *all* `.ts`/`.sh` means a `.ts` file *can* legitimately carry a drift-prone timestamp in a template string — but the wish accepts this tradeoff (those are code, not snapshots). still, it fights the "snapshots" frame: the real target is `.snap`/rendered-output files, yet the rule is phrased by exclusion. a future sharper rule might target *generated* files specifically.
2. **false positives on docs/fixtures**: a markdown changelog line `released at [HH:MM:SS]Z` would be blocked despite intent. without an escape hatch this nags. feels forced.
3. **the message says "in snapshots" but the check isn't snapshot-scoped**: mental-model mismatch — a user blocked on a `.json` fixture sees "forbidden in snapshots" and is confused. either broaden the message or narrow the scope.
4. **regex ambiguity**: `T?HH:MM:SS{timezone}` is informal. `[HH:MM:SS]` also matches durations, sports times, coordinates — decode-friction risk and false positives until the pattern is pinned down precisely.
5. **bypassability**: `--no-verify` defeats it; acceptable for a local guard but means it's a nudge, not a hard guarantee (authoritative enforcement would also need a ci check — possibly out of scope).
