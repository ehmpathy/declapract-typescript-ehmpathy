# vision: slowtest reporter as best practice

## the outcome world

### before

- integration tests run, some take 30+ seconds, nobody notices
- slow tests accumulate silently over weeks/months
- ci/cd build times creep up with no clear culprit
- developers wait longer and longer for feedback
- "why is our build so slow?" becomes a mystery hunt

### after

- every integration test run surfaces a slowtest report
- slow tests (>10s) are immediately visible in terminal output
- json report written to `.slowtest/integration.report.json`
- teams can spot the worst offenders at a glance
- future: ci can use the json for shard optimization

### the "aha" moment

run `npm run test:integration` and see at the bottom:

```
slowtest report:
----------------------------------------------------------------------
🐌 src/domain.operations/syncCustomer.integration.test.ts    32s 150ms [SLOW]
   └── describe: syncCustomerFromStripe (integration)        32s 100ms
       └── given: [case1] customer exists in stripe          32s 50ms
           ├── when: [t0] before sync                        1s 200ms
           │   └── then: phone is null                       1s 200ms
           └── when: [t1] after sync                         30s 850ms
               └── then: phone is updated                    30s 850ms
```

immediately know where to focus optimization efforts.

## user experience

### usecases

| goal | how it works |
|------|--------------|
| spot slow tests locally | run integration tests, see slowtest report in terminal |
| track slow tests in ci | json report written to `.slowtest/integration.report.json` |
| optimize ci shards (future) | consume json report to balance test distribution |

### contract

**input:** none (automatic via jest config)

**output:**
- terminal: formatted slowtest report after test run
- file: `.slowtest/integration.report.json` with structured data

### timeline

1. declapract upgrade → jest.integration.config.ts gets slowtest reporter
2. .gitignore updated → `.slowtest/integration.report.json` ignored
3. test-fns minVersion bumped → 1.7.2 → 1.15.7
4. run integration tests → see slowtest report

## mental model

### how would users describe this?

> "it shows me which integration tests are slow so i can fix them before the build gets out of hand"

### analogies

- like a **performance profiler** but for test duration
- like **lighthouse scores** but for test speed
- like a **fitness tracker** that flags when you slow down

### terms

| user says | we say |
|-----------|--------|
| "slow test alert" | slowtest reporter |
| "test duration report" | slowtest report |
| "which tests are slow?" | threshold: 10s |

## evaluation

### how well does it solve the goals?

| goal | solution quality |
|------|------------------|
| visibility into slow tests | excellent - terminal + json output |
| prevent slow test accumulation | good - makes them visible, still requires action |
| ci optimization | foundation laid - json report enables future shard balancer |

### pros

- zero config for users (comes with declapract best practice)
- non-invasive (doesn't fail tests, just reports)
- actionable (specific test names and durations)
- machine-readable (json for automation)

### cons

- only integration tests (not unit or acceptance)
  - rationale: unit should be fast, acceptance is inherently slow
- threshold is fixed at 10s
  - could be made configurable in future

### pit of success

- gitignore entry prevents accidental commits of report
- integration-only focus keeps unit tests fast by expectation
- json output enables future automation without compatibility breaks

## open questions & assumptions

### assumptions

- 10s is a reasonable threshold for "slow" integration tests
- json report format from test-fns is stable

### design decisions (triaged from open questions)

- **acceptance tests excluded** [answered]: acceptance tests are inherently slow (end-to-end). a slowtest reporter would flag every test. that's noise, not signal.
- **"all tests" means all projects** [answered]: the reference (rhachet-roles-ehmpathy) uses integration-only. "all tests" in the wish means "all projects get the reporter on their integration tests".

### questions for wisher

1. [wisher] is 10s the right threshold, or should it be different?

### research needed

- none required - rhachet-roles-ehmpathy already validated the approach

## what is awkward?

### no major concerns

the implementation is clean:
- test-fns provides the reporter
- jest config adds one entry
- gitignore adds one line
- test-fns minVersion bumps

### minor awkwardness

- threshold is hardcoded to 10s in the best practice
  - acceptable: user can override entire config if different threshold needed
  - plain template file semantics (full overwrite, not CONTAINS)

### exact config change (jest.integration.config.ts)

```ts
reporters: [
  ['default', { summaryThreshold: 0 }],
  [
    'test-fns/slowtest.reporter.jest',
    {
      slow: '10s',
      output: '.slowtest/integration.report.json',
    },
  ],
],
```
