# vision: tune-upgrades

## the outcome world

### day-in-the-life after

a developer upgrades to `declapract-typescript-ehmpathy@0.48.x`:

```bash
npx declapract apply --use-case lambda
```

they see clear, consistent environment names:
- `test` â€” automated verification
- `prep` â€” pre-production (human verification)
- `prod` â€” production

no more "do i need `dev` or `prep`?" confusion. one mental model.

### before/after contrast

| before | after |
|--------|-------|
| `config/dev.json` and `config/prep.json` both referenced | `config/prep.json` only |
| `awsAccountId.dev` in some places, `.prep` in others | `awsAccountId.prep` everywhere |
| `databaseTunnelHost.dev` in shell command but `.prep` in config | `databaseTunnelHost.prep` everywhere |
| `provision/aws/environments/dev/` directory | `provision/aws/environments/prep/` directory (internal `environment = "dev"` preserved) |
| undocumented variables cause cryptic errors | all variables documented with examples |

### the "aha" moment

when a developer defines their variables once:

```yaml
awsAccountId:
  prep: 234567890123  # test environment uses prep account
  prod: 345678901234
```

...and every practice just works. no duplicate keys, no trial-and-error discovery. note: test environment resources are provisioned in the prep AWS account (not a separate account).

### usecases

1. **new repo setup**: developer consults practice readme for required variables, then runs `declapract apply` (note: "fail early on absent variables" is out of scope â€” see evaluation)
2. **upgrade extant repo**: developer sees bad-practice fix that renames `dev` â†’ `prep` automatically
3. **ci/cd config**: variables reference consistent environment names across all practices

### contract inputs

| variable | description | example |
|----------|-------------|---------|

### timeline

1. run `declapract plan` â†’ see violations
2. run `declapract apply` â†’ bad-practices auto-fix config `dev` â†’ `prep`
3. manually rename terraform `environments/dev/` â†’ `environments/prep/` (see migration docs)
4. update variables to use consistent keys (`awsAccountId.{prep,prod}`, `databaseTunnelHost.{prep,prod}`, etc.)
5. run `declapract plan` â†’ clean

**note:** terraform directory rename is manual (cannot be auto-fixed by declapract due to state file references). a new bad-practice `old-terraform-dev-env` could automate detection in a future iteration.

## what is awkward?

### forced tradeoffs

- **breaking change**: repos must update their variable keys from `dev` â†’ `prep`
- **terraform state**: repos with extant `environments/dev/` terraform state will need manual migration (rename state bucket key)

### fights mental model

some teams genuinely call it "dev" not "prep". forcing `prep` may feel foreign initially. however, per `prefer.env_access.prep_over_dev`, the semantic symmetry (`prod`/`prep`/`test` all describe what happens there) outweighs familiarity.

### uncomfortable tradeoffs

terraform state migration is manual and cannot be automated by declapract. we can:
1. document the migration steps clearly
2. provide a bad-practice that warns but doesn't auto-fix (since terraform state is external)
