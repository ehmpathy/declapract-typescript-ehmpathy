# vision: replace joi with zod (and simple-lambda-handlers with sdk-aws-lambda)

## the outcome world

### day-in-the-life

**before:** developer runs `declapract plan` on a repo using joi. no violations appear — joi isn't flagged. the repo continues using a validation library that doesn't integrate well with TypeScript's type system. types are duplicated between joi schemas and interfaces.

**after:** developer runs `declapract plan` on a repo using joi. declapract flags:
1. the `joi` dependency in package.json
2. the `@types/joi` dev dependency
3. every file with `import ... from 'joi'`
4. every file with joi schema patterns (Joi.object, Joi.string, etc.)

running `declapract fix` automatically:
- removes joi/adds zod in package.json
- removes @types/joi
- transforms imports from `import Joi from 'joi'` to `import { z } from 'zod'`
- transforms basic schema patterns from joi syntax to zod syntax

### before/after contrast

| aspect | before (joi) | after (zod) |
|--------|-------------|-------------|
| type inference | manual duplication | automatic via z.infer |
| api style | chained builders | composable functions |
| error messages | generic | rich, structured |
| ecosystem | legacy | modern, active |

### the "aha" moment

when developers realize they no longer need to maintain parallel interface definitions:

```ts
// before: joi — types duplicated
interface Config { name: string; port: number }
const schema = Joi.object({
  name: Joi.string().required(),
  port: Joi.number().required(),
});

// after: zod — single source of truth
const schema = z.object({
  name: z.string(),
  port: z.number(),
});
type Config = z.infer<typeof schema>; // derived automatically
```

## user experience

### usecases

1. **migrate extant joi repos** — run `declapract fix` to automatically convert
2. **enforce zod on new code** — `declapract plan` catches any new joi usage
3. **gradual adoption** — fix violations one at a time or all at once

### contract inputs & outputs

**input:** repo with joi usage patterns:
- `joi` in package.json dependencies
- `@types/joi` in devDependencies
- `import Joi from 'joi'` in source files
- joi schema patterns: `Joi.object({...})`, `Joi.string()`, `Joi.number()`, etc.

**output:** repo with zod equivalents:
- `zod` in package.json dependencies
- zod imports: `import { z } from 'zod'`
- zod schema patterns: `z.object({...})`, `z.string()`, `z.number()`, etc.

### timeline

1. run `declapract plan` — see violations (~30 seconds)
2. run `declapract fix` — apply transforms (~30 seconds)
3. manual review — handle edge cases the transforms couldn't cover (~varies)
4. run tests — verify no regressions

## evaluation

### how well does it solve the goals?

| goal | coverage |
|------|----------|
| remove joi dependency | ✅ 100% — already extant |
| remove @types/joi | ✅ 100% — already extant |
| transform imports | 🟡 high — regex-based |
| transform basic schemas | 🟡 medium — common patterns only |
| transform complex schemas | 🔴 low — manual required |

### pros

- automated migration reduces manual toil
- consistent enforcement across all repos
- fits naturally into extant declapract workflow
- zod has better typescript integration (automatic type inference)

### cons

- regex-based transforms are fragile
- complex joi patterns (custom validators, complex conditionals) need manual migration
- joi-specific features (e.g., `Joi.alternatives()`) have different zod equivalents
- running fix changes many files — can be noisy in PRs

### edgecases and pit of success

| edgecase | handling |
|----------|----------|
| joi in devDependencies | bad practice should check both deps and devDeps |
| multiple import styles | handle `import Joi`, `import * as Joi`, `import { Schema } from 'joi'` |
| schema.validate() vs schema.validateAsync() | both need conversion to zod patterns |
| Joi.alternatives() | flag but don't auto-transform (too complex) |
| nested schemas | basic nesting works, complex refs may need manual |

## open questions & assumptions

### assumptions

1. all repos should migrate to zod (no joi-only use cases)
2. basic schema patterns cover >80% of real-world usage
3. developers can manually handle complex cases after auto-fix
4. zod version 3.24.0+ is the target (already in best practice)

### questions to validate with wisher

1. [answered] **scope:** should we transform schema validation calls too? e.g., `schema.validate(data)` → `schema.parse(data)`?
   - **answer:** no — do not transform validation calls
2. [answered] **depth:** how aggressive should transforms be? basic patterns only, or attempt complex ones?
   - **answer:** basic only
3. [answered] **code that breaks:** is it ok if transforms produce code that doesn't compile (which requires manual fix)?
   - **answer:** yes, no worries — developers can fix manually
4. [answered] **documentation:** do we need migration docs for developers unfamiliar with zod?
   - **answer:** no — not needed

### external research needed

none — zod and joi are well-understood, no external API dependencies.

### impact on scope

| original scope | expanded scope |
|----------------|----------------|
| joi → zod | joi → zod |

### API map

**handler (simple-lambda-handlers → sdk-aws-lambda):**

| aspect | old | new |
|--------|-----|-----|
| function | `createStandardHandler({ log, schema, logic })` | `genLambdaEndpoint({ schema: { input, output }, invoke }, { log })` |
| api gateway | `createApiGatewayHandler({ log, schema, logic, cors })` | `genLambdaEndpoint.for.apiGateway(...)` |
| `logic` arg | `logic: (event) => O` | `invoke: ({ event }, { log }) => O` |
| `schema` arg | `schema` (joi, single) | `schema: { input, output }` (zod, split) |
| `log` arg | passed at creation (required) | passed at creation + received in invoke context |

**client (simple-lambda-client → sdk-aws-lambda):**

| aspect | old | new |
|--------|-----|-----|
| function | `invokeLambdaFunction({ service, function, stage, event })` | `askLambdaEndpoint({ which: { service, function }, event }, context)` |
| stage | `stage: string` | `context.env.access: string` |
| log | `logDebug?: LogMethod` | `context.log: LogMethods` (required) |
| cache | `cache?: SimpleCache<O>` | `context.cache?: { response?, dedupe? }` |
| trail | — | auto-propagated via `context.log` |

**key differences:**
- `logic` → `invoke` (receives `{ log }` in second arg)
- `schema` split into `input` + `output` (both required, zod)
- `stage` → `env.access`
- trail.exid auto-propagated through call chain

## what is awkward?

### regex limitations

transforming `Joi.object({ name: Joi.string() })` to `z.object({ name: z.string() })` with regex works for simple cases, but breaks on:
- nested complex schemas
- multiline schemas
- schemas with comments
- variable references inside schemas

this is a fundamental limitation of regex-based transforms. the alternative (AST-based transforms) would require a different approach than declapract currently supports.

### api differences that don't map 1:1

| joi | zod | awkwardness |
|-----|-----|-------------|
| `Joi.string().required()` | `z.string()` | zod is required by default |
| `Joi.string().optional()` | `z.string().optional()` | maps cleanly |
| `Joi.alternatives().try(a, b)` | `z.union([a, b])` | different structure |
| `Joi.when('field', ...)` | `z.discriminatedUnion(...)` or custom | complex, manual needed |
| `schema.validate(data)` | `schema.parse(data)` or `schema.safeParse(data)` | error handling differs |

### the "manual intervention required" problem

developers might run `declapract fix` and expect all to work. if complex patterns aren't transformed correctly, they'll get compile errors or runtime failures. this is expected but potentially surprising.

**mitigation:** clear documentation in `.declapract.readme.md` explaining scope and limitations.
