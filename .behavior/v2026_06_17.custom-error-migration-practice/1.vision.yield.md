# vision: custom error import migration

## the outcome world

### day-in-the-life with this in place

a developer runs `npx declapract fix` on a repo that has custom error implementations at `src/utils/errors/`. the upgrade:

1. adds `helpful-errors` to package.json (best-practice)
2. deletes `src/utils/errors/UnexpectedCodePathError.ts` (own-implementation bad-practice)
3. **NEW**: rewrites all imports pointing to that deleted file to use `helpful-errors` instead

the developer sees green tests immediately. no manual cleanup required.

### before/after contrast

**before (current state):**

```
# step 1: run declapract fix
$ npx declapract fix

# step 2: run type check
$ npm run test:types
error TS2307: Cannot find module '../../utils/errors/UnexpectedCodePathError'

# step 3: manually fix 47 files with broken imports
# ... 30 minutes of tedious work ...
```

**after (with this bad-practice):**

```
# step 1: run declapract fix
$ npx declapract fix

# step 2: run type check
$ npm run test:types
✓ all types pass

# done. zero manual work.
```

### the "aha" moment

> "wait, it just... fixed all the imports? i didn't have to do anything?"

the value clicks when a developer realizes they saved 30+ minutes of tedious find-and-replace work that they used to do manually after every declapract upgrade.

import { UnexpectedCodePathError } from 'helpful-errors';

## evaluation

### how well does it solve the goals?

| goal | solved? | notes |
|------|---------|-------|
| no manual import fixes | yes | all detected patterns are auto-fixed |
| works with test files | yes | `src/**/*.ts` includes test files |
| handles path variants | yes | relative paths, @src alias paths |

### pros

- **zero friction**: declapract upgrade becomes fully automatic
- **consistent**: uses same pattern as extant `error-fns` bad-practice
- **testable**: unit tests verify check/fix functions
- **composable**: works alongside extant own-implementation bad-practice

### cons

- **regex complexity**: relative path matching has edge cases
- **false positives**: might match non-error files in utils/errors/ (unlikely)
- **import consolidation**: doesn't merge multiple imports into one line (acceptable)

### edgecases and pit-of-success

| edgecase | handling |
|----------|----------|
| import from `utils/errors/index.ts` | matches utils/errors pattern, fixes it |
| import from `utils/error.ts` (singular) | does NOT match, requires different path |
| aliased imports (`import E from ...`) | regex handles import statement variants |
| type-only imports | `import type { X }` also fixed |
| multi-line imports | regex handles with multiline flag |

## open questions & assumptions

### constraints (hard requirements)

1. **only rewrite known error classes**: the fix must ONLY rewrite imports for classes that are exported by helpful-errors:
   - `UnexpectedCodePathError`
   - `BadRequestError`
   - `HelpfulError`
   - `MalfunctionError`
   - `ConstraintError`

   unknown error class names (e.g., `PaymentFailedError`) must be left alone — rewrite would create a broken import.

### assumptions made

1. **repos use standard error class names**: the 5 known classes from helpful-errors
2. **class name at end of path is sufficient**: don't need to match specific directory structures — just `/{KnownClass}` at end
3. **custom classes have different names**: e.g., `UserInputError`, `HumanGuidanceError` — these won't match the allowlist

all questions answered via repo search — no wisher input required:

## groundwork

### external research

**helpful-errors package verified:**
- on npm: `helpful-errors`
- exports used by repos: `UnexpectedCodePathError`, `BadRequestError`, `HelpfulError`, `MalfunctionError`, `ConstraintError`
- best-practice already requires `helpful-errors@^1.7.3`

### internal research

**extant patterns in this repo:**

1. **error-fns bad-practice** at `src/practices/errors/bad-practices/error-fns/`
   - `src/**/*.ts.declapract.ts`: check for `from '@ehmpathy/error-fns'`, fix to `from 'helpful-errors'`
   - test pattern at `src/**/*.ts.declapract.test.ts`
   - this is the template to follow

2. **own-implementation bad-practice** at `src/practices/errors/bad-practices/own-implementation/`
   - `src/utils/errors/UnexpectedCodePathError.ts.declapract.ts`: uses `FileCheckType.EXISTS`
   - deletes the file but doesn't fix imports pointing to it
   - this is the gap being filled

3. **contracts to conform to:**
   - `FileCheckFunction`: `(contents: string | null, context: FileCheckContext) => void`
   - `FileFixFunction`: `(contents: string | null, context: FileFixContext) => { contents?: string | null }`

4. **vocab/naming patterns:**
   - bad-practice directories use kebab-case: `error-fns`, `own-implementation`
   - declapract files use pattern: `{filepath}.declapract.ts`
   - wildcard patterns like `src/**/*.ts.declapract.ts`

## what is awkward?

### regex is simpler than expected

originally worried about path depth, but the pattern is actually simple:
```ts
// match: any local import that ends with /{KnownClass}
// exclude: 'helpful-errors' (already correct)

// pattern: from ['"](?!helpful-errors)[^'"]*\/(UnexpectedCodePathError|BadRequestError|HelpfulError|MalfunctionError|ConstraintError)['"]
```

the class name at the end is the key — we don't need to match specific directory structures.

### naming the bad-practice

options:
- `local-error-imports` — clear but generic
- `own-implementation-imports` — matches extant own-implementation name
- `custom-error-imports` — user-facing term

`own-implementation-imports` feels consistent but long. `local-error-imports` is clearer.

### should this be a separate bad-practice or extend own-implementation?

**separate** (proposed): cleaner separation of concerns — one deletes files, one fixes imports
**combined**: simpler directory structure but mixes file deletion with import rewriting

separate feels cleaner. the own-implementation bad-practice handles file deletion, this new one handles import patterns.

### import consolidation

after the fix, a file might have:
```ts
import { UnexpectedCodePathError } from 'helpful-errors';
import { BadRequestError } from 'helpful-errors';
```

ideally this would become:
```ts
import { UnexpectedCodePathError, BadRequestError } from 'helpful-errors';
```

but that's a separate concern (import organization) and adds complexity. acceptable to leave as-is and let eslint/prettier handle consolidation.
