# vision: github environments for controlled releases

## the outcome world

### day-in-the-life

**before:** a developer merges a PR to main. CI passes. the release workflow runs automatically â€” but production credentials are injected without environment-based gates. anyone who can trigger a workflow can access production secrets. hotfixes from feature branches go through the same ungated flow.

**after:** a developer merges a PR to main. CI passes. the release workflow targets `production-on-main` environment â€” credentials are gated by github's environment protection, but no additional approval is required because the PR merge was the approval gate. if someone needs to deploy from a feature branch (preview, hotfix), they target `production-on-else` â€” which requires explicit reviewer approval before credentials are injected.

### before/after contrast

| aspect | before | after |
|--------|--------|-------|
| release from main | credentials injected without gate | credentials gated by environment (auto-approved via PR) |
| release from feature branch | same ungated credential injection | requires explicit approval before credentials |
| hotfix deployment | no approval gate for credentials | approval gate via production-on-else |
| preview deployment | not standardized | explicit approval, clear audit trail |
| credential access control | workflow trigger = credential access | environment approval = credential access |

### the "aha" moment

> "wait â€” why can anyone who triggers a workflow access production credentials? we need gates."

the value clicks when you realize that github environments provide credential access control:
- **main branch:** PR approval already happened. environment auto-approves credential injection.
- **other branches:** no prior approval. environment requires explicit sign-off before credentials are injected.

## user experience

### usecases

1. **standard release:** developer merges PR to main â†’ automatic deployment with zero additional steps
2. **hotfix release:** developer pushes hotfix branch â†’ requests approval â†’ approved reviewer signs off â†’ deployment proceeds
3. **preview deployment:** developer wants stakeholders to see feature â†’ pushes to branch â†’ requests approval â†’ deploys to preview

### contract inputs & outputs

**inputs (declared in `resources.ts`):**

```typescript
// production-on-main: auto-deploys from main
DeclaredGithubEnvironment.as({
  repo: repoRef,
  name: 'production-on-main',
  reviewers: null,                              // no approval required
  waitTimer: null,                              // no delay
  deploymentBranchPolicy: { customBranches: ['main'] },
  preventSelfReview: false,
});

// production-on-else: requires approval for non-main
DeclaredGithubEnvironment.as({
  repo: repoRef,
  name: 'production-on-else',
  reviewers: { users: ['uladkasach', 'caseybrookes'], teams: null },  // either can approve
  waitTimer: null,                              // no delay once approved
  deploymentBranchPolicy: null,                 // any branch
  preventSelfReview: false,                     // self-approval allowed if in reviewers list
});
```

**outputs (github state):**

- two environments visible in repo settings â†’ environments
- workflows can target these environments
- deployment protection rules enforced by github

### what it looks like to leverage them

```yaml
# .github/workflows/release.yml
jobs:
  release-from-main:
    if: github.ref == 'refs/heads/main'
    environment: production-on-main  # auto-deploys
    steps:
      - run: npm run release

  release-from-else:
    if: github.ref != 'refs/heads/main'
    environment: production-on-else  # requires approval
    steps:
      - run: npm run release
```

## mental model

### how would users describe this to a friend?

> "we have two release gates. if you're on main, you already got approved when the PR merged â€” go straight to production. if you're on any other branch, you need someone to explicitly approve the deployment first."

### analogies

- **building access:** badge in (main) vs visitor sign-in (else). trusted folks go right through.
- **git hooks:** pre-commit (PR approval) vs pre-push (deployment approval). different checkpoints.

## evaluation

### how well does it solve the goals?

| goal | solved? | notes |
|------|---------|-------|
| release PRs merged to main without approval | yes | `reviewers: null` on production-on-main |
| release non-main branches with approval | yes | `reviewers` + `preventSelfReview` on production-on-else |
| clear audit trail | yes | github tracks all deployment approvals |
| prevent accidental production deploys | yes | branch policy restricts which branches can deploy where |

### pros

- eliminates redundant approval for main branch releases
- provides clear approval gate for hotfixes and previews
- leverages github's native environment protection
- declarative â€” provisioned via declastruct, version controlled
- audit trail built into github's deployment history

### cons

- requires `declastruct-github@1.4.0` or later (upgrade needed)
- adds complexity to workflow yaml (conditional environments)
- reviewers must be github users/teams (can't use external systems)

### edgecases and pit of success

| edgecase | how we handle it |
|----------|------------------|
| deploy from main via production-on-else | workflow condition prevents it (`if: github.ref != 'refs/heads/main'`) |
| self-approving a hotfix | allowed â€” `preventSelfReview: false` means reviewers can approve their own deployments |
| no reviewers available | deployment blocks until reviewer responds (mitigated by multiple reviewers) |
| too many reviewers | github limits to 6; declastruct validates |

## what is awkward?

### branch policy ambiguity

the `deploymentBranchPolicy` field has three states:
- `null` = all branches
- `{ protectedBranches: true }` = only protected branches
- `{ customBranches: ['main'] }` = specific branches

for `production-on-else`, we want "all branches EXCEPT main" â€” but there's no direct way to express exclusion. we'd need to either:
- set `null` and rely on workflows to not target production-on-else from main
- explicitly list all allowed branch patterns (fragile)

**resolution:** the workflow itself gates this with `if: github.ref != 'refs/heads/main'`, so the environment's branch policy being `null` is acceptable. the workflow condition is the real gate.

### naming convention

`production-on-main` and `production-on-else` are descriptive but verbose. alternatives considered:
- `prod-main` / `prod-else` (too terse)
- `production` / `production-preview` (misleading â€” else isn't just preview)
- `auto-release` / `approved-release` (describes behavior, not scope)

**resolution:** stick with `production-on-main` and `production-on-else` â€” clarity over brevity.

### reviewer maintenance

~~a single hardcoded reviewer couples infrastructure to one person's availability.~~

**resolved:** wisher specified `['uladkasach', 'caseybrookes']` â€” either can approve, which provides redundancy. self-approval is allowed for those in the list.
