# blocker.1: Global regex replacements in fix functions may corrupt string literals and comments

**rule**: .agent/repo=bhuild/role=behaver/briefs/practices/behavior.execution/architect/rule.forbid.behavior-hazards.md

**locations**:
- src/practices/runtime-type-checking/bad-practices/joi-imports/src/**/*.ts.declapract.ts

The fix functions use global regular expressions (e.g., `replace(/Joi\.string/g, 'z.string')`) that match across the entire file content, including inside string literals and comments. This can silently transform unintended parts of the code, producing incorrect output. For example, a string like `"use Joi.string()"` would become `"use z.string()"`, and a comment containing `.required()` would have that removed. This is a hidden side effect that makes the tool unpredictable for files containing certain patterns. The `.note` acknowledges only basic transforms, but the risk of corrupting valid code is a behavior hazard that can cause hard-to-debug issues after migration.

Consider using AST-aware transformations or at minimum only replacing tokens that are not inside strings/comments.

**snippet**:
```ts
const updated = contents
    // transform imports: import Joi from 'joi' â†’ import { z } from 'zod'
    .replace(/import\s+Joi\s+from\s+['"]joi['"]/g, "import { z } from 'zod'")
    .replace(/import\s+\*\s+as\s+Joi\s+from\s+['"]joi['"]/g, "import { z } from 'zod'")
    .replace(/import\s+\{[^}]*\}\s+from\s+['"]joi['"]/g, "import { z } from 'zod'")
    // transform basic schema patterns: Joi.* â†’ z.*
    .replace(/Joi\.object/g, 'z.object')
    .replace(/Joi\.string/g, 'z.string')
    .replace(/Joi\.number/g, 'z.number')
    .replace(/Joi\.boolean/g, 'z.boolean')
    .replace(/Joi\.array/g, 'z.array')
    .replace(/Joi\.date/g, 'z.date')
    .replace(/Joi\.any/g, 'z.any')
    // transform .required() â†’ remove (zod is required by default)
    .replace(/\.required\(\)/g, '');
```

# nitpick.1: File exports more than one procedure (check and fix)

**rule**: .agent/repo=ehmpathy/role=mechanic/briefs/practices/code.prod/evolvable.procedures/rule.require.single-responsibility.md.min

**locations**:
- src/practices/runtime-type-checking/bad-practices/simple-lambda-handlers/src/**/*.ts.declapract.ts

Each declapract file exports both `check` and `fix` functions, violating the single-responsibility principle that each file should export exactly one named procedure. While these are related, the rule forbids multiple exports per file. Consider splitting into separate files (e.g., one for check logic, one for fix logic) to adhere to the pattern, or providing an explicit exception comment if the pairing is intentional.

**snippet**:
```ts
export const check: FileCheckFunction = (contents) => {
  // ...
};

// ...

export const fix: FileFixFunction = (contents) => {
  // ...
};
```

# nitpick.2: Functions use a positional string argument instead of an object input

**rule**: .agent/repo=ehmpathy/role=mechanic/briefs/practices/code.prod/evolvable.procedures/rule.require.input-context-pattern.md.pt1.md.min

**locations**:
- src/practices/runtime-type-checking/bad-practices/joi-imports/src/**/*.ts.declapract.ts

The `check` and `fix` functions accept a single positional string argument (`contents`) rather than an object input as required by the `(input, context?)` pattern. The rule mandates that all functions take one input object and an optional context object. This makes the code inconsistent with the project's standard contract and reduces readability and extensibility. Consider wrapping `contents` in an object, e.g., `{ contents: string }` as the input, and leaving an optional context second argument even if not needed, to match the convention.

**snippet**:
```ts
export const check: FileCheckFunction = (contents) => {
  // match if file imports from joi
  if (contents?.includes("from 'joi'")) return;
  if (contents?.includes('from "joi"')) return;

  // no match
  throw new Error('does not import from joi');
};
```
