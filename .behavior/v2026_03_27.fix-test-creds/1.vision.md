# vision: keyrack.source() for test credentials

## the outcome world

### day-in-the-life: before

```bash
# developer opens terminal, runs tests
npm run test:integration

# error: apikeys required...
#   run: source .agent/repo=.this/role=any/skills/use.apikeys.sh && npm run test:integration

# developer runs the source command
source .agent/repo=.this/role=any/skills/use.apikeys.sh

# oops, opens new terminal later... repeat the source dance
# every terminal session requires re-source
```

pain points:
- per-terminal auth (forgets to source after new terminal opens)
- error message gives a path to source, but doesn't tell you what keys are absent
- if keys aren't in `~/.config/rhachet/apikeys.env`, error is vague
- no way to know what keys a repo needs without a manual read of the json file

### day-in-the-life: after

```bash
# developer opens terminal, runs tests
npm run test:integration

# if keyrack locked (first time this session):
#   ConstraintError: keyrack not unlocked
#   fix: rhx keyrack unlock --env test --owner ehmpath

# unlock once
rhx keyrack unlock --env test --owner ehmpath

# if keys absent:
#   ConstraintError: keyrack keys absent
#   fixes:
#     - rhx keyrack set --key OPENAI_API_KEY --env test
#     - rhx keyrack set --key ANTHROPIC_API_KEY --env test

# set keys once per machine (only if never set before)
rhx keyrack set --key OPENAI_API_KEY --env test

# now tests work - stays unlocked for the workshift
npm run test:integration  # just works

# open new terminal... still works!
npm run test:integration  # just works
```

benefits:
- once-per-workshift unlock (daemon keeps session alive)
- keys set once per machine, never again
- exact commands to fix issues (no guess work)
- keys are persisted securely (encrypted vault)
- knows what keys exist and what's absent

### the "aha" moment

the "aha" clicks when you open a second terminal and realize you don't need to re-auth. the daemon remembers your session. tests just work.

## user experience

### usecases

| usecase | before | after |
|---------|--------|-------|
| run integration tests | source shell file, then run | just run (keyrack auto-injects) |
| new terminal session | re-source shell file | just run (daemon remembers) |
| discover required keys | read json file manually | error message lists them |
| add a new key | edit env file manually | `rhx keyrack set --key X --env test` |
| ci runs tests | set env vars in workflow | same (keyrack prefers env vars) |

### contract inputs & outputs

**keyrack.source() call:**
```typescript
import { keyrack } from 'rhachet/keyrack';

keyrack.source({
  env: 'test',           // which environment's keys
  owner: 'ehmpath',      // whose keyrack (ssh key owner)
  mode: 'strict',        // fail fast if keys absent
});
```

**keyrack.yml declaration:**
```yaml
org: ehmpathy
env.test:
  - OPENAI_API_KEY
  - ANTHROPIC_API_KEY
```

### declapract scope

changes span two practices:

| practice | change |
|----------|--------|
| **tests** | update jest.integration.env.ts, jest.acceptance.env.ts to use keyrack.source() |
| **tests** | update package.json to remove test:auth command and eval source pattern |
| **cicd-common** | delete use.apikeys.sh, use.apikeys.json and their .declapract.ts files |

note: persist-with-dynamodb jest.integration.env.ts does NOT need changes (no apikeys check there).

### timeline

1. **first time ever on machine**: run `rhx keyrack set` for each key (one-time, persists forever)
2. **first time each workshift**: run `rhx keyrack unlock` (daemon keeps session alive)
3. **rest of workshift**: tests just work, no auth needed

## mental model

### how users describe it

> "keyrack is like a password manager for api keys. unlock it once, and your tests can access the keys they need."

### analogies

| analogy | description |
|---------|-------------|
| password manager | unlock once, autofill everywhere |
| ssh-agent | add key once, use across sessions |
| keychain | secure storage with session unlock |

### terms

| user says | we say |
|-----------|--------|
| "unlock my keys" | `rhx keyrack unlock` |
| "add a key" | `rhx keyrack set --key X` |
| "my keys aren't functional" | keyrack is locked or keys are absent |

## evaluation

### how well does it solve the goals?

| goal | grade | notes |
|------|-------|-------|
| reduce auth friction | A | once per workshift vs once per terminal |
| clear error messages | A | exact commands to fix |
| secure storage | A | encrypted vault, not plaintext env files |
| ci compatibility | A | keyrack prefers env vars first |
| migration simplicity | B | requires rhachet upgrade and file changes |

### pros

- eliminates per-terminal re-auth (unlock once per workshift)
- keys set once per machine, never again
- actionable error messages
- encrypted key storage
- daemon-based session persistence
- ci works without changes (env vars still work)

### cons

- requires rhachet >=1.39.1
- developers must learn `rhx keyrack` commands
- one-time key setup per machine
- one unlock per workshift (but saves many terminal re-auths)

### edge cases and pit of success

| edge case | handle |
|-----------|--------|
| keyrack locked | error with exact unlock command |
| keys absent | error lists each absent key with set command |
| env var already set | keyrack prefers env vars (ci works) |
| no keyrack.yml | keyrack.source() is a no-op |

## what is awkward?

### feels off

- **two files to configure**: both `keyrack.yml` and the jest env file need changes
- **owner parameter**: developers need to know whose keyrack to use (`ehmpath`)

### fights mental model

- developers expect `npm run test` to just work; any auth step feels like friction
- "unlock" implies lock again; developers may worry about re-lock

### uncomfortable tradeoffs

- a daemon dependency for session persistence
- rhachet upgrade required before migration works
- repos without rhachet would break after declapract fix (need migration guidance)
