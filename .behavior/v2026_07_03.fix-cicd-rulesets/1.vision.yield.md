# 1.vision — make `protect-release-tags` ruleset a default best practice

> wish: ehmpathy/declapract-typescript-ehmpathy#515 — elevate the github repo-ruleset
> "protect version tags" to a paved-road default for all ehmpathy typescript repos.

---

## the outcome world

### day-in-the-life

a maintainer spins up (or conforms) a typescript service. they run `declapract fix`.
their `provision/github.repo/resources.ts` now includes a `protect-release-tags`
ruleset alongside the repo, config, branch-protection, and environments they already
had. on the next merge to main, the provision workflow runs `declastruct apply` and
the ruleset materializes on github — `v*` tags become create/update/delete-locked to
the rhelease app only. nobody hand-clicked a github ui ruleset. nobody copy-pasted a
snippet between repos.

### before / after

| dimension | before | after |
|-----------|--------|-------|
| tag protection | per-repo, click-ops in github ui (or absent) | declared once, propagated by declapract |
| prod-apply gate | forgeable — anyone with push can cut/move/delete a `v*` tag | release-app-only, immutable |
| drift | invisible until an incident | a `declastruct plan` CREATE/UPDATE diff, review-gated |
| onboarding a repo | remember to add tag protection by hand | `declapract fix` includes it automatically |

### the "aha"

the value clicks when a maintainer runs `declapract fix` on an old repo and sees the
ruleset *appear for free* — the same way `biome.json` or a workflow file appears. tag
protection stops being a thing you remember to do and becomes a thing the paved road
does for you.

## user experience

### usecases / goals

- **primary**: guarantee that the prod-apply oidc gate ("prod apply only from a version
  tag cut from main") cannot be forged, on *every* repo that releases via rhelease.
- **secondary**: guarantee released version tags are immutable and undeletable
  (reproducibility — a released `v1.2.3` always points at the same commit).
- **tertiary**: zero per-repo effort — one declaration, propagated everywhere.

### contract inputs & outputs

the "contract" here is a **declapract best-practice template file**, not a runtime api.
its inputs are declapract variables already in use; its output is a conformed
`resources.ts`.

| input | source | value |
|-------|--------|-------|
| `@declapract{variable.organizationName}` | declapract var (already used) | e.g. `ehmpathy` |
| `@declapract{variable.projectName}` | declapract var (already used) | e.g. `svc-quotes` |
| rhelease app id | constant (same app org-wide) | `2472031` |
| tag pattern | constant | `refs/tags/v*` |
| rules | constant | `creation`, `update`, `deletion` |
| enforcement | constant | `active` |

output: the extant best-practice `resources.ts` gains a `rulesetVersionTags`
declaration and returns it in the resource set.

### what leveraging it looks like

```ts
// added to src/practices/provision-github/best-practice/provision/github.repo/resources.ts
const rulesetVersionTags = DeclaredGithubRepoRuleset.as({
  repo,
  name: 'protect-release-tags', // release tags are what the rule protects — clearer intent than "version-tags"
  target: 'tag',
  enforcement: 'active',
  bypassActors: [
    { actorId: 2472031, actorType: 'Integration', bypassMode: 'always' }, // rhelease app
  ],
  conditions: { refNameInclude: ['refs/tags/v*'], refNameExclude: [] },
  rules: [{ type: 'creation' }, { type: 'update' }, { type: 'deletion' }],
});

return [ repo, repoConfig, branchMainProtection, teamReleasersAccess,
         envProductionOnMain, envProductionOnElse, rulesetVersionTags ];
```

### timeline

1. this repo: add the declaration to the best-practice template + bump the
   `declastruct-github` minVersion so the symbol exists.
2. release declapract-typescript-ehmpathy.
3. each downstream repo, on its next `declapract fix` / conform pass, gains the ruleset
   in `resources.ts`.
4. that repo's provision workflow applies it via `declastruct apply` on merge to main.

## mental model

### how a user describes it to a friend

> "our best-practices tool now ships version-tag protection by default — you can't
> fake a release tag anymore, only the release bot can cut `v*` tags, and once cut they
> can't be moved or deleted."

### analogies

- **a wax seal on a release**: once the release bot stamps `v1.2.3`, the seal can't be
  re-forged or peeled off.
- **the paved road**: like every other declapract default (biome, tsconfig, workflows),
  you don't opt in — the road already has the guardrail.

### their terms vs our terms

| they say | we say |
|----------|--------|
| "lock the release tags" | `DeclaredGithubRepoRuleset` with `creation`/`update`/`deletion` rules |
| "only the release bot" | `bypassActors: [{ actorType: 'Integration', ... }]` |
| "the `v*` tags" | `conditions.refNameInclude: ['refs/tags/v*']` |
| "make it the default" | add to the `best-practice/` variant so `declapract fix` propagates it |

## evaluation

### how well it solves the goals

- **strongly** for the primary goal: the oidc prod gate becomes un-forgeable on every
  repo, with no per-repo effort — exactly the "paved-road default, not per-repo
  copy-paste" the wish asks for.
- the declaration is a **verbatim lift** of the already-dogfooded, live ruleset on
  `ehmpathy/declastruct-github` (id 18380440), so behavior is proven, not speculative.

### pros

- one source of truth; drift shows as a reviewable `declastruct plan` diff.
- verbatim reuse of a live, tested declaration → low risk.
- fits the extant resource-set shape in the same file → minimal surface area.

### cons / tradeoffs

- **requires `admin`/`administration` repo scope** on each target's provision — the
  conformer app already grants this (per the wish), but any repo whose token lacks it
  will fail the apply.
- **hardcoded app id `2472031`** — correct today (rhelease is a public app shared
  org-wide) but a magic number; documented via comment.
- **byte-for-byte EQUALS check** — this best-practice file is conformed exactly, so any
  repo that customized `resources.ts` will see this as an overwrite. (this is already
  true for the file today; we're not changing the check semantics.)

### edgecases & pit-of-success

| edgecase | how the contract keeps us safe |
|----------|-------------------------------|
| repo doesn't release via rhelease | ruleset still harmless — it only blocks `v*` tag writes by non-rhelease actors; repos that never cut `v*` tags are unaffected |
| app id drift (rhelease reinstalled) | surfaces as a `declastruct plan` diff on bypass actor; caught in review |
| someone customized `resources.ts` | EQUALS conform overwrites — expected declapract behavior; diff is reviewable pre-apply |
| target token lacks admin scope | apply fails loudly at provision time (not silently) |

## open questions & assumptions

### assumptions

1. **the rhelease app id `2472031` is org-wide constant** across all ehmpathy repos
   (the wish states this; the live ruleset confirms it for declastruct-github).
2. **the conformer app already grants `administration` scope** on target repos (per the
   wish's notes) — so no scope change is needed here.
3. **the best-practice `resources.ts` is conformed via EQUALS** (no `.declapract.ts`
   companion exists for it) → adding the declaration propagates it as-is. *(verified:
   no companion file in the best-practice dir.)*

### resolved during groundwork

1. **✅ ruleset export first published in `1.6.0`.** installed `declastruct-github` is
   **`1.4.0`**, which does **not** export the symbol (verified by grep of `node_modules`).
   the export first appears in **`v1.6.0`** — verified against the published tags:
   `src/contract/sdks/index.ts` at tag **`v1.5.7`** has **no** ruleset export (28 lines);
   at tag **`v1.6.0`** line 22 is `export { DeclaredGithubRepoRuleset }`. so `1.6.0` is
   the correct minimum floor (not the earlier 1.5.7 patch). execution must:
   - bump the best-practice `package.json` pin from `minVersion('1.5.5')` → `minVersion('1.6.0')`
   - add `DeclaredGithubRepoRuleset` to the template's import block from `'declastruct-github'`
     (the current file imports the peer entities there but not this one yet).
   - **note**: no change to *this* repo's own deps — the template isn't typechecked here
     and `declastruct-github` isn't a direct dep (see triaged open questions).

## groundwork

### external research

- **github repo rulesets** — real feature; the declaration mirrors the github rulesets
  api (`target`, `enforcement`, `bypass_actors`, `conditions.ref_name.include/exclude`,
  `rules[].type`). verified indirectly via the declastruct-github domain object which
  maps 1:1 to the api (`src/domain.objects/DeclaredGithubRepoRuleset.ts`).
- **rhelease app id `2472031`** — asserted by the wish and matched by the live ruleset;
  not independently re-queried (`gh api /apps/rhelease` would confirm, deferred as
  low-risk since it's a verbatim lift).

### internal research (all cited)

- **target file**: `src/practices/provision-github/best-practice/provision/github.repo/resources.ts`
  — extant resource-set builder; returns `[repo, repoConfig, branchMainProtection,
  teamReleasersAccess, envProductionOnMain, envProductionOnElse]` (lines 158–166). the
  ruleset appends to this list.
- **reference implementation**: `ehmpathy/declastruct-github` →
  `provision/github.repo/resources.ts:141–165` — the live, dogfooded `protect-version-tags`
  declaration. our template lifts the values with two deliberate changes: `owner/name`
  swapped for declapract variables, and the ruleset **renamed `protect-version-tags` →
  `protect-release-tags`** (clearer intent: the rule protects *release* tags). note: `name`
  is part of the unique key (`['repo','name']`), so on declastruct-github's own repo a
  conform would create a new `protect-release-tags` ruleset beside the extant
  `protect-version-tags` — a one-time migration to reconcile (retire the old name) at
  execution, tracked as a [wisher]/[research] follow-up.
- **domain object contract**: `ehmpathy/declastruct-github` →
  `src/domain.objects/DeclaredGithubRepoRuleset.ts` — `unique = ['repo','name']`,
  `primary = ['id']`, nested `bypassActors`/`conditions`/`rules` as `DomainLiteral`s.
  `.as({ repo, name, target, enforcement, bypassActors, conditions, rules })`.
- **check semantics**: no `resources.ts.declapract.ts` companion exists in the
  best-practice dir → default `FileCheckType.EQUALS` (per `declapract-default-check-equals`
  brief) → conform overwrites the whole file.
- **dep pin**: `src/practices/provision-github/best-practice/package.json` pins
  `declastruct-github: @declapract{check.minVersion('1.5.5')}` — must rise to the
  ruleset-exporting version.
- **wiring**: `src/useCases.yml` — `provision-github` is included by `npm-package` (l29),
  `lambda-service` (l45), and `app-react-native-expo` (l74); `lambda-service-with-rds`
  and `lambda-service-with-dynamodb` inherit it by extending `lambda-service`.
- **vocab / style**: matches the extant declaration idioms in the same file (`.as({...})`,
  lowercase comments, `.why` rationale comments).

## what is awkward?

- **the version dependency is the crux, not the declaration.** the code change is ~15
  lines of verbatim lift; the real work is the best-practice `package.json` pin bump to
  `minVersion('1.6.0')` so downstream repos install a version with the symbol. now
  confirmed unblocked — `1.6.0` is published and exports the ruleset. (this repo's own
  deps are untouched — the template isn't typechecked here.)
- **magic app id `2472031`** hardcoded in a template that fans out to every repo — it's
  correct but a single point of "if rhelease is ever reinstalled with a new id, every
  repo drifts at once." mitigated only by the review-gated `declastruct plan` diff.
- **EQUALS overwrite of `resources.ts`** means any repo that legitimately customized its
  provision resources will fight the conform. this tension already exists for this file;
  we inherit it rather than introduce it.
- **no dedicated test surface** — best-practice template files aren't unit-tested here
  the way `.declapract.ts` fix functions are; correctness rides on the verbatim lift +
  the upstream declastruct-github test suite. verification will lean on typecheck +
  a conform plan-run rather than a new test.
- **return-array order** — the file has an explicit order-dependency
  (`teamReleasersAccess` "must come before environments that reference this team").
  the ruleset only refs `repo` (first in the array), so it's dependency-safe to append
  last — it matches the reference impl, which also returns it last. no order hazard,
  but worth a conscious check rather than a blind append.
- **repo-level vs org-level** — a single `DeclaredGithubOrgRuleset` on the ehmpathy org
  would protect `v*` tags across all repos in one declaration (DRYer than N repo
  rulesets). but declastruct-github v1 ships repo rulesets only (org deferred upstream),
  and org rulesets need `admin:org` scope. so repo-level is the available choice today;
  a future consolidation to org-level is the cleaner end-state once upstream ships it.
