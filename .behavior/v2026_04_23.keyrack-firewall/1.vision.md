# 1.vision — keyrack firewall upgrade

upgrade from adhoc secrets parse to `rhx keyrack firewall` for CI workflow secrets injection.

**scope**: all workflow types that use keyrack secrets (test.yml, deploy.yml, etc.)

**note on `$GITHUB_ENV`**: firewall exports to `$GITHUB_ENV`, which is job-scoped. each job that needs secrets must run the firewall step.

---

## the outcome world

### before: adhoc secrets parse

```ts
// buildWorkflowSecretsBlock.ts — manual slug parse
const keyrackVars = keys.map((k) =>
  (k.status === 'granted' ? k.grant.slug : k.slug).split('.').pop()!,
);

// build secrets block manually
const secretsBlock = keyrackVars
  .map((key) => `      ${key}: \${{ secrets.${key} }}`)
  .join('\n');
```

declapract generates workflow YAML with secrets blocks. each project declares keys in `keyrack.yml`, then declapract:
1. calls `keyrack.get()` to fetch keys
2. parses slugs to extract key names (format: `org.env.KEY_NAME`)
3. builds `secrets:` declaration block for workflow_call
4. builds `env:` block for test jobs

the slug parse is adhoc — it assumes format, handles status variants, and breaks if the format changes.

### after: keyrack firewall

```yaml
# .test.yml — workflow passes secrets to firewall
- name: keyrack firewall
  run: npx rhachet keyrack firewall --env test --from 'json(env://SECRETS_JSON)' --into github.actions
  env:
    SECRETS_JSON: ${{ toJSON(secrets) }}
```

the firewall:
1. reads `keyrack.yml` to know which keys belong to this env
2. filters input to only declared keys (security boundary)
3. translates mechanisms (GitHub App → ghs_* token)
4. validates patterns (blocks ghp_*, AKIA*)
5. exports to `$GITHUB_ENV` with proper masking

no more adhoc parse. the firewall handles key extraction, filtering, and sanitization.

### the aha moment

**before**: declapract has to know keyrack's slug format and parse it correctly.
**after**: declapract just says "run the firewall" — keyrack knows its own format.

the complexity moves from consumer (declapract) to owner (rhachet). when keyrack's format changes, only keyrack updates — not every consumer.

## user experience

### usecases

| goal | before | after |
|------|--------|-------|
| add key to test env | update keyrack.yml, run declapract fix, verify generated YAML | update keyrack.yml — firewall auto-includes it |
| translate GitHub App | not supported here (handled elsewhere) | firewall translates automatically |
| block dangerous tokens | not supported | firewall blocks ghp_*, AKIA*, gho_*, ghu_*, ghr_* |
| debug missing key | read generated YAML, trace back to keyrack.get() | `npx rhachet keyrack firewall --env test --from ... --into json` |

### contract inputs & outputs

**input**: `${{ toJSON(secrets) }}` — GitHub's secrets context as JSON
**output**: secrets exported to `$GITHUB_ENV` with `::add-mask::` for each value

```yaml
# caller workflow (test.yml)
jobs:
  test:
    uses: ./.github/workflows/.test.yml
    secrets: inherit  # or explicit secrets block
```

```yaml
# callee workflow (.test.yml)
- name: keyrack firewall
  run: npx rhachet keyrack firewall --env test --from 'json(env://SECRETS_JSON)' --into github.actions
  env:
    SECRETS_JSON: ${{ toJSON(secrets) }}
```

### timeline

1. `declapract plan` — see the diff (secrets blocks removed, firewall step added)
2. `declapract fix` — apply changes
3. push — CI runs with firewall
4. firewall validates keys, exports to env
5. tests run with credentials in `$GITHUB_ENV`

## evaluation

### how well does it solve the goals?

| goal | score | notes |
|------|-------|-------|
| remove adhoc parse | ✅ | slug parse moves to rhachet |
| simplify declapract | ✅ | refactor buildWorkflowSecretsBlock.ts, simplify workflow declapract files |
| add security validation | ✅ | blocks ghp_*, AKIA*, etc. |
| add mechanism translation | ✅ | GitHub App → ghs_* token |
| maintain backwards compat | ✅ | extant workflows work after fix |

### pros

- **single source of truth**: keyrack.yml declares keys, firewall handles the rest
- **security**: validates patterns, blocks dangerous tokens
- **mechanism translation**: short-lived tokens from long-lived credentials
- **maintainability**: fewer lines of code in declapract
- **debugging**: `--into json` mode for troubleshooting

### cons

- **new dependency**: requires rhachet version with firewall command
- **workflow change**: need to add firewall step to all test workflows
- **migration**: extant projects need `declapract fix` to update

### edgecases

| edgecase | handling |
|----------|----------|
| project has no keyrack.yml | firewall step is no-op (no keys to process) |
| key declared but not in secrets | status=locked, not an error |
| dangerous token in secrets | status=blocked, exit code 2, CI fails |
| mechanism translation fails | status=locked, error logged |

## what is awkward?

### passing secrets as JSON

```yaml
env:
  SECRETS_JSON: ${{ toJSON(secrets) }}
```

this feels verbose. ideally:

```yaml
- run: npx rhachet keyrack firewall --env test --into github.actions
  # (auto-reads from secrets context)
```

but GitHub Actions doesn't allow direct secrets access in `run:` — must pass via `env:`. this is a platform constraint, not a design flaw.

### firewall step in every workflow

every test workflow needs the firewall step. this is repetitive but:
- makes the security checkpoint explicit
- enables per-workflow customization (different envs)
- follows the "explicit is better than implicit" principle

### migration path

extant projects have generated secrets blocks. after migration:
- caller workflows may still have `secrets:` blocks (harmless)
- callee workflows use firewall instead of env vars

this is a one-time cleanup via `declapract fix`.

## summary

upgrade from adhoc secrets parse to keyrack firewall:

1. **refactor** `buildWorkflowSecretsBlock.ts` — replace slug parse with firewall invocation, or delete if no longer needed
2. **simplify** workflow declapract files — remove slug parse logic
3. **update** `.test.yml`, `.deploy.yml`, etc. — add firewall step to each job that needs secrets
4. **remove** secrets declaration/env blocks from generated YAML (use `secrets: inherit` in callers)

the firewall handles key filter, mechanism translation, and pattern validation. declapract becomes simpler; security gets stronger.
